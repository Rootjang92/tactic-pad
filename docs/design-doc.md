## Problem Statement

스포츠 코치들은 전술과 훈련 세션을 계획/시각화/전달하기 위해 전술 보드 앱을 사용한다. 시장 1위인 TacticalPad(2010년 출시, 250만 다운로드)는 15년간 UX/성능이 정체되어 있고, 3D 전환이 불편하며, 전반적인 사용 경험이 낮다. 불만이 있음에도 대안이 없어 계속 사용 중인 코치들이 존재한다.

핵심 기회: 60개 이상의 스타트업이 AI/웨어러블/VR에 집중하는 동안, 코치가 매일 여는 기본 도구인 전술 보드 자체는 아무도 제대로 다시 만들지 않았다.

## Demand Evidence

- K리그 프로팀 피지컬 코치: "3D 전환이 매우 불편하고, 전체적인 사용성과 성능이 너무 낮다"
- 앱스토어/리뷰 공통 불만: 크래시, 동기화 문제, 기기 간 데이터 접근 불가, 폴더/정리 기능 부재
- 코치들이 불만을 가지면서도 TacticalPad를 계속 사용 (대안 부재 = 잠금 효과)
- 증거 수준: 약함 (1명 코치의 고수준 피드백). 프로토타입 기반 추가 검증 필수.

## Status Quo

K리그 피지컬 코치는 TacticalPad를 계속 사용 중. 불만에도 불구하고 대안이 없음.

일반적인 코치 워크플로우(검증 필요):

- 전술 보드에서 포메이션/드릴을 그림
- 애니메이션으로 선수 이동 시각화
- 결과물을 선수에게 전달 (방법 미확인: 라이브 프레젠테이션? 영상? 이미지? 링크?)

## Target User & Narrowest Wedge

**타겟:** 한국 축구 코치 (K리그 프로 + 아마추어/유소년)

- 피지컬 코치, 전술 코치, 감독
- 매일 훈련 계획을 세우고 선수에게 전달하는 사람

**가장 좁은 쐐기:**
축구 반 코트 위에서 선수 토큰을 드래그하고, 키프레임 애니메이션으로 전술 이동을 시각화하는 단일 기능. 설치 없이 브라우저에서 바로. 이것이 전술 보드의 "원자적 가치 단위"다.

## Constraints

1. 웹 기반 (Next.js PWA). MVP(Phase 3)에서 오프라인 퍼스트 구현 — 훈련장/라커룸에서도 동작해야 함
2. 한국 시장 우선. 한국어 UI는 Phase 1부터 적용 (constraint, not hypothesis)
3. 코치의 실제 워크플로우를 아직 모름 — 프로토타입으로 관찰 후 설계 확정
4. 1인 개발 (추정)
5. SaaS 구독 모델 ($5-10/월), TacticalPad ($63/년) 대비 낮은 진입점

## Premises

1. 핵심 문제는 TacticalPad의 UX/성능이 2010년 수준에 멈춰있다는 것. 기본 도구 품질 > AI 등 신기능.
2. 초기 타겟은 한국 축구 코치 (프로 + 아마추어). 한국 시장 먼저.
3. **수정됨:** 웹 기반(Next.js PWA). Phase 1은 온라인 전용 프로토타입, Phase 3에서 오프라인 퍼스트(Service Worker + IndexedDB) 구현. 세컨드 오피니언의 도전을 수용 — 코치는 인터넷이 불안정한 환경(경기장, 훈련장)에서 사용하므로 MVP 출시 시점에는 오프라인 필수.
4. SaaS 구독 ($5-10/월).
5. 현재 수요 증거는 약함. 프로토타입 기반 추가 검증 필수.

## Cross-Model Perspective

독립 AI(Claude subagent)의 콜드 리드:

**스틸맨:** TacticalPad가 250만 다운로드로 수요를 증명했고, 60+ 스타트업이 AI/VR에 몰리는 동안 전술 보드 자체는 비어있다. 한국 축구에 특화된 모던 웹 도구가 틈새를 잡을 수 있다.

**핵심 인사이트:** "코치의 워크플로우를 모른다"는 것이 가장 중요한 신호. 라이브 도구, 내보내기 도구, 프레젠테이션 도구, 화이트보드 — 이건 각각 다른 제품이다. 이걸 확인하지 않으면 모든 설계 결정이 추측이다.

**도전된 전제:** 웹 기반은 경기장/라커룸 환경에서 위험. PWA + 오프라인으로 해결 가능하지만 "단순한 웹사이트" 접근은 안 된다. → 전제 수정 수용됨.

**48시간 프로토타입:** Next.js + Konva.js, 반 코트 캔버스 + 드래그 토큰 + 키프레임 애니메이션 + URL 공유. 3D/비디오/멀티스포츠/계정 모두 스킵.

## Approaches Considered

### Approach A: 미니멀 전술 보드

축구 반 코트 + 드래그 토큰 + 키프레임 애니메이션. localStorage + URL 공유. 계정/DB 없음.

- Effort: S (1-2주) | Risk: Low | Completeness: 4/10

### Approach B: 풀 MVP

전술 보드 + 프로젝트/폴더 + Supabase + PWA 오프라인 + 이미지/GIF 내보내기 + 팀 관리.

- Effort: M (4-6주) | Risk: Medium | Completeness: 7/10

### Approach C: 관찰 먼저, 빌드 나중 (선택됨)

Approach A 프로토타입(1주) → 코치 3-5명 관찰(2주) → 관찰 기반 풀 MVP(3주).

- Effort: S+M (6주) | Risk: Low | Completeness: 9/10

## Recommended Approach

**Approach C: 관찰 먼저, 빌드 나중**

이유:

1. 가장 큰 리스크(코치 워크플로우 미확인)를 정면으로 해결
2. 프로토타입 → 관찰 → 설계 순서로 헛수고 최소화
3. 관찰 대상 코치들이 자연스럽게 초기 사용자가 됨
4. Approach A 코드가 Approach B의 기반이 되므로 낭비 없음

### Phase 1: 프로토타입 (1주)

**Tech Stack:** Next.js 16 + Konva.js (2D 캔버스) + Tailwind 4. 한국어 UI.

- Konva.js 모바일 성능이 부족할 경우 대안: Fabric.js 또는 raw Canvas 2D API.

**기능:**

- 축구 반 코트 캔버스 (녹색 필드, 라인)
- 드래그 가능한 선수 토큰 (번호 원형, 2색 팀 구분, 최대 22명 + 공 1개)
- **키프레임 애니메이션 타임라인:**
  - 최대 10개 키프레임. UI 스크러버(타임라인 바)로 키프레임 간 이동.
  - 각 키프레임에서 토큰 위치를 기록. 재생 시 linear interpolation으로 이동.
  - 재생 속도: 1x (기본), 0.5x, 2x 토글.
  - 키프레임 추가/삭제 가능. 순서 변경은 Phase 3으로 연기.
  - 재생 시 이동 궤적(점선 trail) 표시.
  - 공 토큰도 선수 토큰과 동일하게 독립적으로 드래그/키프레임 기록. 토큰 간 부모-자식 관계 없음.
- **공유:** localStorage에 저장 + JSON export/import. URL 공유는 state가 커질 수 있으므로(22토큰 x 10키프레임 = ~2KB+) URL 파라미터 대신 base64 인코딩된 JSON을 짧은 hash로 localStorage에 저장하고 share ID로 URL 생성. 프로토타입 단계에서는 백엔드 없이 같은 기기 내 공유만 지원. 기기 간 공유는 JSON 파일 다운로드/업로드로 대체.
- 모바일/태블릿 반응형. 터치 드래그 지원.
- **엣지 케이스:** 지원하지 않는 브라우저(IE 등)는 "Chrome/Safari를 사용해주세요" 안내 표시.

**스킵:** 3D, 비디오, 멀티스포츠, 계정, 폴더, AI, 이미지/GIF 내보내기, 오프라인, 기기 간 실시간 공유

### Phase 2: 코치 관찰 (2주)

- K리그 피지컬 코치 포함 3-5명 코치에게 프로토타입 전달. **최소 2명 확보 시 진행.** 5명 미만일 경우 원격 세션(화면 공유 + 녹화)으로 보완.
- "써보세요" 후 옆에서 관찰 (가이드 없이)
- 핵심 관찰 항목:
  - 코치가 그린 전술을 어떻게 전달하는가? (라이브? 영상? 이미지? 링크?)
  - 어디서 쓰는가? (사무실? 라커룸? 훈련장?)
  - 무엇을 먼저 하는가? 어디서 막히는가?
  - "이것도 되면 좋겠다"고 말하는 게 있는가?

### Phase 3: 풀 MVP 설계 & 빌드 (3주)

- Phase 2 관찰 결과 기반으로 설계 확정
- **확정 기능 (premises에서 도출):**
  - PWA 오프라인 퍼스트 (Service Worker + IndexedDB)
  - Supabase 백엔드 + 계정 + 기기 간 동기화
- **예상 추가 기능 (관찰 전 가설, Phase 2에서 검증):**
  - 프로젝트/폴더 관리
  - 이미지/GIF 내보내기
  - 팀 관리
- **수익화:** 프리미엄 모델. 기본 전술 보드는 무료 (토큰 + 3키프레임). 유료($5-10/월)로 무제한 키프레임, 내보내기, 프로젝트 폴더, 클라우드 동기화. 결제: Stripe 또는 Toss Payments(한국). 구체적 기능 분류는 Phase 2 관찰 후 확정.
- **오프라인-온라인 동기화 충돌 전략:** Phase 3 설계 시 확정. 최소 접근: last-write-wins (타임스탬프 기반). 3주 타임라인이 부족할 경우 '예상 추가 기능'(폴더, 내보내기, 팀관리)을 Phase 4로 이동하고 Phase 3은 오프라인 + 계정 + 동기화에 집중.

## Open Questions

1. **코치가 전술을 선수에게 어떻게 전달하는가?** 이것이 제품의 형태를 결정한다 (라이브 도구 vs 내보내기 도구 vs 프레젠테이션 도구).
2. **3D가 정말 필요한가?** 코치의 "3D 전환 불편" 불만이 "3D 자체는 필요한데 구현이 나쁘다"인지, "3D가 불필요한 단계인데 강제된다"인지 확인 필요.
3. **태블릿 vs 데스크톱?** 코치가 주로 어떤 기기에서 작업하는지.
4. **한국 아마추어/유소년 코치 시장 규모?** K리그 프로팀은 소수. 실제 볼륨은 아마추어.
5. **경쟁사 Tactico가 이미 브라우저 기반.** 차별점은 무엇인가? (한국어 + 한국 축구 특화?)

## Success Criteria

1. Phase 2 관찰에서 3명 이상의 코치가 프로토타입을 자발적으로 "다시 쓰고 싶다"고 말함
2. 코치가 프로토타입으로 만든 전술을 실제 선수에게 보여주는 장면을 목격
3. Phase 3 MVP 출시 후 30일 내 유료 전환 사용자 5명 이상
4. 코치 1명이 기존 TacticalPad에서 완전 전환

## Distribution Plan

- **접근 채널:** 웹 (PWA) — 설치 없이 URL로 바로 접근
- **배포:** Vercel (Next.js 네이티브 호스팅)
- **CI/CD:** GitHub Actions → Vercel 자동 배포 (main 브랜치 머지 시)
- **초기 유통:** K리그 코치 네트워크 통한 직접 공유 + 한국 축구 커뮤니티 (에펨코리아, 각종 축구 카페)

## Dependencies

1. **코치 네트워크:** Phase 2 관찰을 위해 3-5명 코치 섭외 필요. K리그 피지컬 코치가 연결고리. 최소 2명 확보 시 진행, 미달 시 원격 세션으로 보완.
2. **Konva.js 검증:** 캔버스 성능이 모바일/태블릿에서 충분한지 Phase 1 첫째 날에 확인. 성능 부족 시 대안: Fabric.js → raw Canvas 2D API 순서로 전환.
3. **PWA 오프라인:** Service Worker + IndexedDB 설계는 Phase 3에서 구현. Phase 1은 온라인 전용.

## The Assignment

이번 주에 해야 할 한 가지:

**K리그 피지컬 코치에게 연락해서 30분 미팅을 잡으세요.** 프로토타입이 아직 없어도 괜찮습니다. 미팅 목적은 다음 3가지 질문에 답을 얻는 것:

1. "TacticalPad로 전술을 그린 다음에 선수들에게 어떻게 보여주세요?" (전달 방식)
2. "주로 어디서, 어떤 기기에서 쓰세요?" (사용 환경)
3. "가장 자주 하는 작업이 뭐예요? 매일 반복하는 거요." (핵심 워크플로우)

이 3개 답을 가지고 오면 프로토타입 설계가 추측이 아니라 사실에 기반하게 됩니다.

## What I noticed about how you think

- 당신은 실존하는 프로 코치의 피드백에서 출발했습니다. "이런 앱이 있으면 좋겠다"가 아니라 "이 사람이 이렇게 말했다"에서 시작한 것. 이건 중요합니다.
- 증거가 약하다는 전제 #5에 동의한 것. "내 아이디어가 완벽하다"고 우기지 않고 현실을 인정하는 건 좋은 파운더의 특성입니다.
- 세컨드 오피니언이 웹 기반 전제를 도전했을 때, 전제를 수정했습니다. "오프라인 퍼스트 추가"라는 구체적 수정이었지 전체를 뒤집은 게 아님. 적절한 판단.
- Approach C(관찰 먼저)를 선택한 것. 빠르게 만들고 싶은 유혹이 있었을 텐데, "모르는 것을 먼저 알아내자"를 선택했습니다.
